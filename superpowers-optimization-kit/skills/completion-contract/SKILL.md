# Completion Contract — 完成契约活文档

## 概述

完成契约是全自动交付的**硬门槛活文档**：从骨架起步，随阶段门禁推进更新，最终用四项硬门槛决定能否宣布完成。本 skill 规定初始化时机、保存路径、阶段更新、硬门槛检查步骤、验收场景与已试方法规则，以及更新后的固定输出格式。

原则（对应设计决策 D2–D9、D12）：

1. **用户 PRD 永远最高** — 可细化，不可改目标
2. **宣布完成仅当硬门槛全绿** — 禁止口头宣称
3. **验收场景可新增，禁止删除/降级已冻结行**
4. **已试方法只追加** — 禁止重复相同方法摘要（防原地踏步）

## 何时使用

- Intake 门禁通过后初始化正式活文档
- Pre-grill / goal-document 产出 Goal 时创建契约骨架
- 每个阶段门禁通过后更新对应字段与证据
- auto-recovery 失败后追加「已试方法」
- 完成门前执行硬门槛检查
- 任何时刻需要回答「现在能否宣布完成」

**不适用：**

- 尚未有 PRD / Goal 骨架时凭空宣称完成
- 用对话摘要代替契约文件作为完成证据

## 路径与文件约定

| 项 | 约定 |
|----|------|
| 保存路径 | `docs/superpowers/contracts/YYYY-MM-DD-<topic>-contract.md` |
| 目录不存在 | 先创建 `docs/superpowers/contracts/` |
| 模板来源 | 优先复制 `superpowers-optimization-kit/templates/completion-contract.md`；若尚未安装模板，按下方「文档结构」手写等价结构 |
| 与 Goal 文档关系 | Goal 文档内可内嵌完成契约章节；**权威副本**始终是 `contracts/` 下独立文件，阶段更新以 contracts 文件为准 |
| topic | 小写英文/数字/连字符，来自任务名或 PRD 主题（如 `health-endpoint`） |
| 日期 | 初始化当日 `YYYY-MM-DD`；之后更新不改文件名 |

## 初始化时机

按以下顺序选择最早适用的时机：

1. **骨架创建（可选更早）**：Pre-grill 产出 Goal 文档时，同步创建 contracts 骨架（状态多为 pending，`可否宣布完成=否`）
2. **正式跟踪起点（必须）**：Intake 门禁通过后，契约进入正式跟踪；验收场景在 Intake 冻结后不得删除/降级

初始化步骤：

1. 确认 PRD 路径与（若有）Goal 文档路径
2. 创建 `docs/superpowers/contracts/`（如不存在）
3. 写入 `docs/superpowers/contracts/YYYY-MM-DD-<topic>-contract.md`
4. 从 PRD 填入 **PRD 目标清单**（覆盖状态=`pending`，证据路径空）
5. 从 PRD 验收 + P1/wiki 填入 **验收场景**（状态=`pending`，冻结=`骨架` 或 Intake 后改为 `是`）
6. 填决策矩阵 / 未决项（可空表）
7. 硬门槛四项全部未勾选；`可否宣布完成=否`
8. 输出「=== 完成契约更新 ===」块

## 文档结构（权威字段）

契约文件必须包含以下章节（顺序固定）：

```markdown
# 完成契约：{任务名称}

## 状态
- 当前阶段：
- 可否宣布完成：否
- 最后更新：
- 关联 Goal：
- 关联 PRD：

## PRD 目标清单
| # | 目标 | 来源 | 覆盖状态 | 证据路径 |
|---|------|------|---------|---------|

## 验收场景
| # | 场景 | 类型 | 证据要求 | 状态 | 证据路径 | 冻结 |
|---|------|------|---------|------|---------|------|

## 决策矩阵
| # | 决策点 | 决策 | 来源 | 置信度 |
|---|--------|------|------|--------|

## 未决项
| # | 问题 | 状态 | 处理方式 |
|---|------|------|---------|

## 已试方法（禁止原地踏步）
| # | 失败门禁 | 方法摘要 | 结果 | 时间 |
|---|---------|---------|------|------|

## 硬门槛
- [ ] PRD 目标全部覆盖
- [ ] 验收场景全部有实测证据
- [ ] 无未决冲突
- [ ] Wiki/检查点合规

## 变更日志
| 时间 | 阶段 | 变更摘要 |
|------|------|---------|
```

字段取值约定：

| 字段 | 允许值 |
|------|--------|
| 覆盖状态 / 场景状态 | `pending` / `in_progress` / `done` / `blocked` |
| 未决项状态 | `open` / `resolved` / `accepted-risk`（仅非冲突类） |
| 冻结 | `骨架` / `是` / `后增` |
| 可否宣布完成 | `是` / `否` |
| 当前阶段 | `Intake` / `Spec+Plan` / `实现` / `验收证据` / `Wiki/合规` / `完成门` / `已完成` |

模板顶部规则注释（若手写文件也须遵守）：

- 验收场景：可新增隐含缺口；禁止删除/降级已冻结场景
- PRD 目标不可改（不可删行、不可改目标语义）
- 宣布完成仅当硬门槛全绿

## 更新时机

每个**阶段门禁通过后**必须更新契约（门禁 FAIL 时只允许追加已试方法 / 未决项，不得把门槛标绿）。

| 阶段通过后 | 至少更新 |
|------------|----------|
| Intake | 目标清单完整；验收场景冻结（`冻结=是`）；决策矩阵；未决冲突清空或处理 |
| Spec+Plan | 证据路径可指向 spec/plan；相关目标 `in_progress` |
| 实现 | 实现相关证据路径；无新增 open 冲突 |
| 验收证据 | 每条场景 `状态=done` + 真实证据路径 |
| Wiki/合规 | 硬门槛第 4 项证据（wiki-checkpoint 报告路径） |
| 完成门 | 跑完整硬门槛检查；仅全绿时 `可否宣布完成=是` |

每次更新：

1. 读现有契约全文
2. 按规则改字段（见下）
3. 写回同一路径
4. 追加变更日志一行
5. 输出「=== 完成契约更新 ===」块

## 验收场景规则

1. **可新增**：发现 PRD 隐含缺口时追加新行；新行 `冻结=后增`，状态从 `pending` 起
2. **禁止删除**已存在行（含 `冻结=是` / `后增` / 已 `done` 的行）
3. **禁止降级**：不得把 `done` 改回 `pending` 来「放宽」验收；不得削弱「证据要求」语义
4. **冻结后**（Intake 通过）：原骨架行 `冻结=是`，之后只允许改状态/证据路径，不允许改场景含义或类型以逃避验收
5. 证据必须是**可核查路径**（测试输出文件、curl 日志、截图、门禁报告等），禁止填「已验证」而无文件

## 已试方法规则

由 `auto-recovery`（或等价恢复流程）在**每次失败后**追加：

1. 读「已试方法」表全部「方法摘要」
2. 新策略的方法摘要**不得**与任一已有摘要相同（禁止原地踏步）
3. 追加一行：失败门禁、方法摘要、结果、时间
4. **只追加不删改**历史行
5. 策略穷尽时：契约保持 `可否宣布完成=否`，另写失败报告（`docs/superpowers/failures/...`），不伪造全绿

## 硬门槛检查（函数式步骤 — 必须逐步执行）

完成门或任何「是否可宣布完成」判断时，**禁止凭记忆口头通过**。按下列步骤执行：

### 步骤 A：读契约

1. 打开权威契约文件：`docs/superpowers/contracts/YYYY-MM-DD-<topic>-contract.md`
2. 若文件不存在 → 硬门槛 0/4，`可否宣布完成=否`，停止并先初始化

### 步骤 B：PRD 目标全部覆盖

对「PRD 目标清单」**每一行**：

1. `覆盖状态` 必须为 `done`
2. `证据路径` 非空
3. 证据路径指向的文件必须存在（用工具检查，不假设）

任一目标不满足 → 门槛 1 红。

### 步骤 C：验收场景全部有实测证据

对「验收场景」**每一行**：

1. `状态` 必须为 `done`
2. `证据路径` 非空且文件存在
3. 不得忽略 `冻结=后增` 的行

任一场景不满足 → 门槛 2 红。

### 步骤 D：无未决冲突

检查「未决项」表：

1. 不存在 `状态=open` 的冲突/矛盾项（问题描述含冲突、矛盾、blocker、不可调和等，或处理方式标明为冲突类）
2. 更严规则：完成门时**任意** `状态=open` 的未决项均视为不通过（默认采用此规则）

存在 open 项 → 门槛 3 红。

### 步骤 E：Wiki/检查点合规

1. 契约或关联证据中必须有 **wiki-checkpoint 通过** 的记录
2. 报告路径存在且内容总体为通过（如「总体：✅ 通过」或等价 PASS）
3. 将该路径写入硬门槛第 4 项相关证据（可写在变更日志或场景/目标证据中）

无通过报告 → 门槛 4 红。

### 步骤 F：汇总与宣布完成

```
硬门槛得分 = 通过项数 / 4
```

| 条件 | 动作 |
|------|------|
| 4/4 全绿 | 勾选硬门槛四项；`可否宣布完成` 改为 **是**；`当前阶段` 可为 `已完成` |
| 任一红 | 硬门槛对应项保持未勾选；`可否宣布完成` 必须为 **否**；**禁止**向 goal judge / 用户宣布完成 |

**宣布完成仅当四项全绿。** 部分绿不得折中放行。

## 阶段更新检查清单（门禁通过后）

执行 Agent 在 gatekeeper 返回 PASS 且报告文件存在后：

- [ ] 读契约
- [ ] 更新 `当前阶段` 与 `最后更新`
- [ ] 回写本阶段相关目标/场景状态与证据路径
- [ ] 确认未引入 `open` 冲突（若有则写入未决项，不得标完成）
- [ ] 追加变更日志
- [ ] 输出固定格式块

完成门额外：

- [ ] 完整执行步骤 A–F
- [ ] 仅 4/4 时改 `可否宣布完成=是`

## 输出格式（强制）

每次初始化或更新契约后，必须在对话中输出：

```
=== 完成契约更新 ===
路径：docs/superpowers/contracts/YYYY-MM-DD-<topic>-contract.md
当前阶段：{阶段名}
硬门槛：{通过数}/4
可否宣布完成：是/否
```

可选附加一行摘要（不替代上述块）：

```
变更：{一句话}
```

## 与其它 skill 的协作

| Skill | 关系 |
|-------|------|
| `goal-document` | 生成 Goal 时创建/内嵌契约骨架，并复制权威副本到 `contracts/` |
| `pre-grill` | 可行后触发 Goal；可触发骨架创建 |
| `gatekeeper` | 各阶段 PASS 后由执行 Agent 更新契约；完成门检查 `硬门槛 4/4` 与 `可否宣布完成=是` |
| `auto-recovery` | 失败后追加已试方法；不得在未换策略时重复摘要 |
| `decision-arbiter` | 矛盾处理后更新决策矩阵与未决项（open→resolved） |
| `wiki-checkpoint` | 通过报告路径写入契约，作为硬门槛第 4 项证据 |

## 红线

以下行为视为违规，必须立即纠正：

1. 无契约文件或硬门槛未 4/4 时宣布完成
2. 删除或降级已冻结/已有验收场景
3. 修改 PRD 目标语义或删除目标行
4. 证据路径填空话且文件不存在
5. 重复相同「方法摘要」却声称换了策略
6. 执行 Agent 自检放行完成门（完成门仍须 gatekeeper + 本 skill 硬门槛步骤）
7. 把 Goal 内嵌章节当唯一权威却不更新 `contracts/` 副本

## 验证清单

- [ ] 契约位于 `docs/superpowers/contracts/YYYY-MM-DD-<topic>-contract.md`
- [ ] 含状态 / PRD 目标 / 验收场景 / 决策矩阵 / 未决项 / 已试方法 / 硬门槛 / 变更日志
- [ ] 硬门槛检查逐步执行且证据文件真实存在
- [ ] `可否宣布完成=是` 当且仅当硬门槛 4/4
- [ ] 更新后输出 `=== 完成契约更新 ===` 块
- [ ] 验收场景无删除/降级；新增行为 `冻结=后增`
- [ ] 已试方法仅追加且摘要不重复
