# L2-架构 — 系统怎么搭的

> 读完知道"系统怎么搭的"。技术选型、模块划分、数据流的全景图。

## 系统架构

```
采集层（仅 GitHub）：
  一次性 bulk：initial_collection.py ──> data/raw/github-initial/ + 安全 merge → projects.yaml
  每日增量：collect_github.py ──> data/raw/github/YYYY-MM-DD/  (JSON 快照)
  （Exa / fallback-web 脚本保留但已停用）

处理层：
  raw/ ──> normalize.py ──> data/projects.yaml         (全量归一化索引库)
  data/projects.yaml ──> score.py ──> 100 分制（60 可量化 + 40 LLM 占位/质量）
  score.py ──> finalize_data.py ──> curated / rejected
  每周：weekly_analysis.py ──> LLM 质量分 + tracking_priority 分级

报告层：
  data/ + tools/ ──> generate_reports.py ──> docs/reports/*.md

站点层：
  data/ + reports/ ──> build_site.py ──> site/  (静态站点源码 + 数据 JSON)
  site/ ──> deploy_site.py ──> /var/www/coding.lzpgood.online/ (Nginx 部署，opt-in)

版本控制层：
  data/ docs/ site/ VERSION CHANGELOG ──> git commit ──> GitHub 总仓库
  GitHub ──> Actions: Update Data / Publish Site / Create Release

调度层：
  Hermes cron 每日 03:00 增量采集；每周一 03:30 LLM；每周一 09:00 release
  GitHub Actions 仓库侧验证（无生产 --deploy 除非显式）
```

## 技术选型

| 技术 | 理由 |
|------|------|
| Python 3.12+ | 脚本语言，生态丰富，适合数据处理 pipeline |
| GitHub CLI (gh) | 原生搜索工具，无需额外 API key，复用 GitHub token |
| Exa via mcporter | 首选语义搜索源，质量高于通用 web 搜索 |
| YAML 数据存储 | 人类可读、Git 友好、JSON 兼容 |
| 静态站点（纯 HTML+JS） | 无需后端服务，安全且成本低 |
| Nginx | 高性能静态文件服务器，支持安全响应头 |
| Let's Encrypt | 免费 TLS 证书，自动续期 |
| GitHub Actions | 仓库侧自动化，利用 GitHub 环境运行 |
| GitHub Pages | 免费静态站点预览发布 |
| Hermes Agent | 生产环境调度器，支持 cron、skills、subagent |
| 日期版本号 | 简单直观，适合数据持续更新的项目 |
| 无数据库依赖 | 全部数据以 YAML/JSON 文件存储，Git 作为版本数据库 |

## 模块划分

### 采集模块
- `initial_collection.py` — 一次性历史 bulk（stars 分层 L1–L5b + code search + 安全 merge + checkpoint）
- `collect_github.py` — 每日增量：`gh search repos` + `gh repo view`
- `collect_exa.py` / `collect_web.py` — 保留停用（不再写入 projects）

### 数据处理模块
- `normalize.py` — 原始数据归一化为统一格式，分类规则驱动
- `score.py` — 100 分制（60 可量化 + 40 LLM 质量）
- `finalize_data.py` — 自动生成 curated/rejected 数据集
- `validate_data.py` — 数据模式验证
- `weekly_analysis.py` / `llm_api.py` / `benchmark_manager.py` — 每周 LLM 质量分析
- `enrich_i18n.py` / `enrich_translations.py` / `translate_summaries.py` — 中英文双语增强

### 报告与构建模块
- `generate_reports.py` — 生成 10+ 份 Markdown 报告
- `build_site.py` — 构建站点 JSON 数据 + 复制报告
- `deploy_site.py` — 部署到 Nginx webroot

### 工具模块
- `common.py` — 公共工具函数（文件 I/O、评分、字段推断）
- `quality_gate.py` — 质量门禁检查
- `sanitize_public_data.py` — 数据脱敏
- `snapshot_and_diff.py` — 快照与差异比较
- `archive_raw.py` — 原始数据归档
- `bump_version.py` — 版本号升级
- `release_notes.py` — 发布说明生成

### 测试模块
- `test_pipeline_features.py` — 管道功能测试
- `test_data_integrity.py` — 数据完整性测试

### 前端模块 (site/)
- `index.html` — SPA 入口骨架
- `app.js` — 全部前端逻辑（数据加载、渲染、搜索、筛选、排序、双语）
- `styles.css` — 深色主题样式
- `data/` — 构建生成的 JSON 数据
- `reports/` — 预生成的 Markdown 报告

## 数据流全景

1. **（一次性）历史 bulk**：`initial_collection.py` 分层搜索 → 详情 → 安全 merge 进 `projects.yaml`
2. **（每日）增量采集**：`update_tracker.py` → `collect_github.py` → `normalize.py`
3. **评分**：100 分制可量化分每日更新；LLM 质量分每周更新
4. **终结**：根据评分自动生成 curated / rejected
5. **报告**：生成 curated-top / tool-comparison / weekly-report 等
6. **构建**：YAML → `site/data/*.json`，复制报告到 `site/reports/`
7. **质量门禁**：完整性、覆盖指标、official-seed track 等
8. **部署**：显式 `--deploy` 才同步 Nginx webroot
9. **版本控制**：git commit；push 由维护者/cron 策略决定

## 部署架构

- **生产环境**：Nginx 服务器（腾讯云上海），/var/www/coding.lzpgood.online/
- **TLS**：Let's Encrypt 自动续期，仅支持 TLSv1.2/TLSv1.3
- **安全头**：HSTS、CSP 严格模式、X-Content-Type-Options、X-Frame-Options、Permissions-Policy 等
- **GitHub Pages**：作为预览发布面，通过 Actions 自动部署
- **调度**：Hermes cron 负责生产部署，GitHub Actions 只负责仓库侧验证

## 关键边界

- **前端 vs 后端**：前端只负责展示，所有数据在后端 pipeline 中预生成
- **Mock vs 真实**：无 mock 模式，所有数据来自真实采集
- **仅 GitHub 源**：生产 projects 仅 `github` + `official-seed`；Exa/web 历史 raw 保留不入库
- **Official-seed 保护**：`source_type=official-seed` 强制 `tracking_priority=track`
- **部署职责分离**：生产 deploy 必须显式 `--deploy`；GitHub Actions 默认不做生产同步
