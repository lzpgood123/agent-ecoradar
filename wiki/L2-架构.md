# L2-架构 — 系统怎么搭的

> 读完知道"系统怎么搭的"。技术选型、模块划分、数据流的全景图。

## 系统架构

```
采集层：
  GitHub Search (gh) ──> data/raw/github/YYYY-MM-DD/  (JSON 快照)
  Exa (mcporter)     ──> data/raw/exa/YYYY-MM-DD/     (JSON 快照)
  fallback-web       ──> data/raw/web/YYYY-MM-DD/     (JSON 快照)

处理层：
  raw/ ──> normalize.py ──> data/projects.yaml         (全量归一化索引库)
  data/projects.yaml ──> score.py ──> data/scores.yaml (评分结果)
  score.py ──> finalize_data.py ──> data/curated-projects.yaml (推荐集)
                                  ──> data/rejected-projects.yaml (噪声集)

报告层：
  data/ + tools/ ──> generate_reports.py ──> docs/reports/*.md (10+ 份报告)

站点层：
  data/ + reports/ ──> build_site.py ──> site/  (静态站点源码 + 数据 JSON)
  site/ ──> deploy_site.py ──> /var/www/coding.lzpgood.online/ (Nginx 部署)

版本控制层：
  所有 data/ docs/ site/ VERSION CHANGELOG ──> git commit ──> GitHub 总仓库
  GitHub ──> Actions: Update Data / Publish Site / Create Release

调度层：
  Hermes cron（每日 03:00 运行 update_tracker.py）
  GitHub Actions（每日 cron 运行 update-data.yml，无 --deploy）
```

## 技术选型

| 技术 | 理由 |
|------|------|
| Python 3.12+ | 脚本语言，生态丰富，适合数据处理 pipeline |
| GitHub CLI (gh) | 原生搜索工具，无需额外 API key，复用 GitHub token |
| Exa via mcporter | 首选语义搜索源，质量高于通用 web 搜索 |
| YAML 数据存储 | 人类可读、Git 友好、JSON 兼容 |
| 静态站点（纯 HTML+JS） | 无需后端服务，安全且成本低 |
| Nginx | 高性能静态文件服务器，支持安全响应头 |
| Let's Encrypt | 免费 TLS 证书，自动续期 |
| GitHub Actions | 仓库侧自动化，利用 GitHub 环境运行 |
| GitHub Pages | 免费静态站点预览发布 |
| Hermes Agent | 生产环境调度器，支持 cron、skills、subagent |
| 日期版本号 | 简单直观，适合数据持续更新的项目 |
| 无数据库依赖 | 全部数据以 YAML/JSON 文件存储，Git 作为版本数据库 |

## 模块划分

### 采集模块 (scripts/collect_*.py)
- `collect_github.py` — 通过 `gh search repos` 和 `gh repo view` 搜索 GitHub
- `collect_exa.py` — 通过 mcporter 调用 Exa 语义搜索
- `collect_web.py` — 通用 web 搜索 fallback

### 数据处理模块
- `normalize.py` — 原始数据归一化为统一格式，分类规则驱动
- `score.py` — 6 维度评分 + 源权重/分类权重/惩罚
- `finalize_data.py` — 自动生成 curated/rejected 数据集
- `validate_data.py` — 数据模式验证
- `enrich_i18n.py` / `enrich_translations.py` — 中英文双语增强

### 报告与构建模块
- `generate_reports.py` — 生成 10+ 份 Markdown 报告
- `build_site.py` — 构建站点 JSON 数据 + 复制报告
- `deploy_site.py` — 部署到 Nginx webroot

### 工具模块
- `common.py` — 公共工具函数（文件 I/O、评分、字段推断）
- `quality_gate.py` — 质量门禁检查
- `sanitize_public_data.py` — 数据脱敏
- `snapshot_and_diff.py` — 快照与差异比较
- `archive_raw.py` — 原始数据归档
- `bump_version.py` — 版本号升级
- `release_notes.py` — 发布说明生成

### 测试模块
- `test_pipeline_features.py` — 管道功能测试
- `test_data_integrity.py` — 数据完整性测试

### 前端模块 (site/)
- `index.html` — SPA 入口骨架
- `app.js` — 全部前端逻辑（数据加载、渲染、搜索、筛选、排序、双语）
- `styles.css` — 深色主题样式
- `data/` — 构建生成的 JSON 数据
- `reports/` — 预生成的 Markdown 报告

## 数据流全景

1. **采集**：Hermes cron 调用 `update_tracker.py`，依次执行 collect_github、collect_exa、normalize
2. **归一化**：将原始 JSON 快照读取并合并为统一的项目记录，按 URL 去重
3. **评分**：对每个项目 6 维度评分，加上源权重、分类权重、惩罚值，计算 total_score
4. **终结**：根据评分自动生成 curated 推荐集和 rejected 噪声集
5. **报告**：读取评分后数据，生成各类 Markdown 报告
6. **构建**：将 YAML 数据转换为 JSON 写入 site/data/，复制报告到 site/reports/
7. **质量门禁**：检查数据完整性、覆盖指标、i18n 完整性
8. **部署**：将 site/ 同步到 Nginx webroot
9. **版本控制**：git commit + push 回 GitHub 总仓库

## 部署架构

- **生产环境**：Nginx 服务器（腾讯云上海），/var/www/coding.lzpgood.online/
- **TLS**：Let's Encrypt 自动续期，仅支持 TLSv1.2/TLSv1.3
- **安全头**：HSTS、CSP 严格模式、X-Content-Type-Options、X-Frame-Options、Permissions-Policy 等
- **GitHub Pages**：作为预览发布面，通过 Actions 自动部署
- **调度**：Hermes cron 负责生产部署，GitHub Actions 只负责仓库侧验证

## 关键边界

- **前端 vs 后端**：前端只负责展示，所有数据在后端 pipeline 中预生成
- **Mock vs 真实**：无 mock 模式，所有数据来自真实采集
- **GitHub vs Exa**：GitHub 数据 source_quality=verified，Exa 数据=verified，fallback-web=fallback
- **Official vs Ecosystem**：官方工具永远单独展示（ranking_scope=official），不进入生态排名
- **部署职责分离**：Hermes cron 负责生产部署（--deploy），GitHub Actions 只做仓库侧验证
