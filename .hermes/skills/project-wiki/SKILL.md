# 项目 Wiki 构建器

## 概述

一次性构建项目全息手册（`wiki/` 目录），通过 3 个子 Agent 并行分析项目代码和文档，填充 11 层分层文档（8 核心 + 3 规划），并将维护规则写入 HERMES.md。之后由 HERMES.md 规则自动维护。

## 何时使用

- 新项目初始化，需要建立文档体系
- 旧项目想建立 wiki 手册，让新 Agent 快速理解项目
- 用户说"构建 wiki"、"初始化项目手册"、"生成项目文档"
- 项目缺少 `wiki/` 目录

**不适用：** wiki 已存在且只需增量更新（由 HERMES.md 规则处理，无需加载本技能）

## 执行流程

```
加载技能 → 检测 wiki/ → 不存在则初始化 → 创建骨架 → 派子Agent分析 → 填充 → 写入HERMES.md
```

### 步骤 1：检测项目状态

检查 `wiki/` 目录是否存在：
- **不存在** → 进入初始化流程（步骤 2-4）
- **存在** → 询问用户："wiki 已存在，是否重建？（会覆盖现有内容）"

### 步骤 2：创建 wiki 骨架

创建 `wiki/` 目录，写入 11 个文档文件。每个文件先写入章节标题模板（见下方模板），内容留空，等待子 Agent 分析结果填充。

**重要：** 必须一次性创建全部 11 个文件，确保子 Agent 返回后能直接填充。

### 步骤 3：派 3 个子 Agent 并行分析

**必须并行分派，不得串行。** 3 个子 Agent 的职责和范围：

| 子 Agent | 职责 | 分析范围 |
|----------|------|---------|
| Agent A | 前端代码分析 | `frontend/`、`src/`、`public/`、`components/`、`pages/` 等前端目录 |
| Agent B | 后端代码分析 | `backend/`、`server/`、`api/`、`routes/`、`services/` 等后端目录 |
| Agent C | 项目文档分析 | `docs/`、`README.md`、`CONTEXT.md`、`HERMES.md`、`.env.example`、ADR |

**分派提示词模板：**

```
## Agent A — 前端代码分析

全面分析 {项目路径} 下的前端代码文件。你需要读取所有前端文件并返回详细分析。

对每个文件返回：文件路径、行数、职责（一句话）、所有关键函数/类的名称和用途、与其他文件的依赖关系、关键数据结构。

对所有前端代码返回总结：
- 页面/路由列表和对应的渲染函数
- 页面状态流转逻辑
- 状态管理结构（state 对象完整定义）
- 样式体系（CSS 变量、主题、设计模式）
- 关键交互模式（表单提交、按钮点击、页面切换）
- 任何特殊设计决策

请确保返回所有细节，不要省略任何函数名、变量名或数据字段。
```

```
## Agent B — 后端代码分析

全面分析 {项目路径} 下的后端代码文件。你需要读取所有后端文件并返回详细分析。

对每个文件返回：文件路径、行数、职责（一句话）、所有关键函数/类的名称和用途、与其他文件的依赖关系（import 关系）、关键数据结构。

对所有后端代码返回总结：
- 所有 API 端点的完整清单（路径、方法、请求体、响应体、错误码）
- 数据校验规则（如适用）
- 错误处理机制（try/except 位置、兜底策略）
- 配置管理（环境变量、配置文件加载逻辑）
- 测试覆盖情况

请确保返回所有细节，不要省略任何函数名、类名、路由路径或数据字段。
```

```
## Agent C — 项目文档分析

全面分析 {项目路径} 下的所有项目文档。你需要读取以下文件：
- README.md、CONTEXT.md、HERMES.md
- docs/ 目录下所有文件
- .env.example
- 任何 ADR 或设计文档

返回以下总结：
- 产品一句话描述 + 核心价值主张
- 用户的核心使用流程
- 核心功能列表（每个功能 2-3 行描述）
- 当前项目状态（已完成、进行中、待办）
- 系统架构概览
- 技术选型及其理由
- 所有关键设计决策摘要
- 已知坑和注意事项
- 环境配置项说明

请确保返回所有细节，尤其是产品功能描述和设计决策。
```

### 步骤 4：填充文档 + 写入 HERMES.md

**填充顺序：** 先填 README.md（总索引），再按 L1→L6→P1→P2→P3 顺序填充。每个文档填充时参考对应子 Agent 的分析结果。

**填充后写入 HERMES.md：** 检查 HERMES.md 中是否已有"项目手册（Wiki）"章节。如有则跳过，如无则在"Agent skills"章节之前插入（见下方规则文本）。

**完成后 commit：**
```bash
git add wiki/ HERMES.md
git commit -m "docs: 初始化Wiki项目全息手册（11个文档）"
```

## 文档约定

- 每个文档末尾有"下一步读什么"章节，指向下一层文档
- 每个文档有"更新指引"章节，说明何时更新、如何更新
- 标注 `[动态]` 的章节需在每次阶段结束时更新

## 快速链接

{11 个文档的链接列表}

## 更新指引

**何时更新：** 每个阶段完成时、发现新坑时、代码结构变更时、接口变更时
**更新原则：** 谁改代码谁更新 wiki，只更新变更的部分

## 常见问题

### 项目没有前后端分离怎么办？

模板中的"前端"和"后端"是逻辑概念。对于单体项目：
- L4A-前端详解 → 面向 UI 层面（页面、组件、样式）
- L4B-后端详解 → 面向服务层面（API、数据库、业务逻辑）
- 如果只有一侧，L4 层只生成一个文档

### 项目没有文档怎么办？

Agent C 只分析 README.md 和 HERMES.md（如有）。L1-全景、L6-经验录中缺少的内容标注"待补充"。

### wiki 已存在但内容过时怎么办？

本技能检测到 wiki/ 存在后，询问用户。用户选择"重建"则覆盖，选择"取消"则退出（由 HERMES.md 规则处理增量更新）。
